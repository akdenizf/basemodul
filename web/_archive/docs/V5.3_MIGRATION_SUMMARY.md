# Callfolio v5.3 - Async Architecture Migration

## Was wurde geändert?

### Problem (v5.2)
Das alte System hatte einen fundamentalen Design-Fehler:
1. E-Mails wurden **direkt im Vapi Webhook** versendet
2. Die E-Mail enthielt **Rohdaten** (was die KI gehört hat: "Fatig Akinew")
3. Das **Fuzzy Matching** fand erst später im Dashboard statt
4. Wenn die E-Mail versendet wurde, aber der DB-Insert fehlschlug, gab es eine **Benachrichtigung über ein nicht-existierendes Ticket**

### Lösung (v5.3)
Wir haben die Architektur komplett umgestellt:

```
ALT (v5.2):
Vapi Call → Webhook → [DB Insert + E-Mail senden] → Response

NEU (v5.3):
Vapi Call → Webhook → [DB Insert] → Response
                          ↓
                    DB Trigger
                          ↓
              Async Processor → [Fuzzy Match + E-Mail]
```

## Geänderte Dateien

### 1. `/app/api/vapi/webhook/route.ts`
**Änderungen:**
- ❌ Entfernt: Alle `Resend` Imports und E-Mail-Logik
- ❌ Entfernt: `notification_email` Abfragen
- ✅ Hinzugefügt: Strikte Fehlerbehandlung (DB-Insert schlägt fehl → Anrufer wird informiert)
- ✅ Hinzugefügt: Detailliertes Logging für jeden Schritt
- ✅ Verbessert: `raw_caller_name` und `raw_caller_address` werden korrekt befüllt

**Vorher (v5.2):**
```typescript
// E-Mail direkt im Webhook
if (dbSuccess && ticketId && notificationEmail) {
  await resend.emails.send({ ... });
}
```

**Nachher (v5.3):**
```typescript
// Nur DB-Insert, keine E-Mail
console.log(`📤 RESPONSE: code=${ticketCode}`);
console.log(`   NOTE: Email will be sent asynchronously after fuzzy matching`);
return NextResponse.json({ ... });
```

### 2. `/app/api/tickets/process-new/route.ts` (NEU)
**Zweck:** Asynchroner Processor für Fuzzy Matching und E-Mail-Versand

**Workflow:**
1. Ticket aus DB laden
2. Fuzzy Matching durchführen:
   - Telefonnummer-Match (exakt, Last 10 Digits)
   - Name/Adresse-Match (via `match_tenant` RPC, Score ≥ 0.7)
3. Ticket mit korrekten Daten aktualisieren
4. E-Mail mit sauberen Daten versenden
5. Activity Log schreiben

**Match-Typen:**
- `PHONE_EXACT`: Telefonnummer stimmt exakt → Auto-Assign
- `FUZZY_HIGH`: Name/Adresse Score ≥ 0.7 → Auto-Assign
- `FUZZY_LOW`: Score 0.4-0.69 → Manuelle Review nötig
- `NONE`: Kein Match → Gast-Ticket

### 3. `/database/add_async_fields_to_tickets.sql` (NEU)
**Zweck:** Migration für neue Felder in `tickets` Tabelle

**Neue Felder:**
- `match_type` (TEXT): Typ des Matchings
- `processed_at` (TIMESTAMPTZ): Wann wurde Fuzzy Matching durchgeführt?
- `email_sent_at` (TIMESTAMPTZ): Wann wurde E-Mail versendet?

**Zweck:** Vollständige Anleitung für Setup und Troubleshooting

**Inhalt:**

2. Create new hook:
   - **Name**: `process-new-ticket`
   - **Table**: `tickets`
   - **Events**: `INSERT`

## Vorteile der neuen Architektur

### 1. Daten-Integrität ✅
- **Problem**: E-Mail wurde versendet, auch wenn DB-Insert fehlschlug
- **Lösung**: E-Mail wird nur versendet, wenn Ticket in DB ist

### 2. Saubere Daten ✅
- **Problem**: E-Mail enthielt "Fatig Akinew" (KI-Transkription)
- **Lösung**: E-Mail enthält "Fatih Akdeniz" (nach Fuzzy Matching)

### 3. Fehler-Isolation ✅
- **Problem**: Wenn Resend down ist, hängt der Vapi-Call
- **Lösung**: Vapi-Call ist sofort fertig, E-Mail wird asynchron versucht

### 4. Retry-Logik ✅
- **Problem**: Fehlgeschlagene E-Mails wurden nicht erneut versucht
- **Lösung**: Supabase Webhooks haben automatisches Retry (5x mit Exponential Backoff)

### 5. Audit Trail ✅
- **Problem**: Keine Transparenz über Matching-Qualität
- **Lösung**: `match_type` zeigt genau, wie gut der Mieter identifiziert wurde

## Breaking Changes

### Für Entwickler
- `Resend` Import wurde aus `webhook/route.ts` entfernt
- `notification_email` wird nicht mehr im Webhook abgefragt
- Neue API-Route: `/api/tickets/process-new`

### Für Betreiber
- **Supabase Database Webhook muss manuell eingerichtet werden**
- Alte Tickets (vor v5.3) haben `processed_at = NULL` und müssen ggf. nachverarbeitet werden

### Für Endnutzer
- **Keine Breaking Changes** - Das System funktioniert identisch, nur zuverlässiger

Cron-Job für Tickets, bei denen E-Mail fehlgeschlagen ist:
```typescript
const { data: failedTickets } = await supabase
  .from('tickets')
  .select('id')
  .is('processed_at', null)
  .lt('created_at', new Date(Date.now() - 5 * 60 * 1000));
```

### 3. Statistiken
```sql
SELECT 
  match_type, 
  COUNT(*) as count,
  AVG(EXTRACT(EPOCH FROM (email_sent_at - created_at))) as avg_time
FROM tickets
WHERE processed_at IS NOT NULL
GROUP BY match_type;
```
